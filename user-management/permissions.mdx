## Overview

Redash uses a multi-layered permission system:

1. **User Permissions:** Group-based permissions that control what actions users can perform
2. **Object Permissions:** Individual access grants for queries and dashboards
3. **Data Source Permissions:** Group-level access to data sources with view-only options

## Permission Model

### Group-Based Permissions

Users inherit permissions from all groups they belong to. Permissions are additive - if a user belongs to multiple groups, they have all permissions from all groups.

**Permission storage** (from `/home/daytona/workspace/source/redash/models/users.py:173-175`):

## Permission Checks

### Decorator-Based Checks

Permissions are enforced using decorators on API endpoints:

**Single permission:**

```python
@require_permission("admin")
def post(self):
    # Only admins can access
```

**Multiple permissions (all required):**

```python
@require_permissions(("create_query", "execute_query"))
def post(self):
    # User must have both permissions
```

**Multiple permissions (any one required):**

```python
@require_any_of_permission(["admin", "super_admin"])
def post(self):
    # User must have at least one permission
```

**Implementation** (from `/home/daytona/workspace/source/redash/permissions.py:57-84`):

```python
class require_permissions:
    def __init__(self, permissions, allow_one=False):
        self.permissions = permissions
        self.allow_one = allow_one
    
    def __call__(self, fn):
        @functools.wraps(fn)
        def decorated(*args, **kwargs):
            if self.allow_one:
                has_permissions = any([current_user.has_permission(permission) for permission in self.permissions])
            else:
                has_permissions = current_user.has_permissions(self.permissions)
            
            if has_permissions:
                return fn(*args, **kwargs)
            else:
                abort(403)
        
        return decorated
```

## Data Source Permissions

### Access Control Model

Data sources use a group-based access control model:

1. **Groups are granted access** to data sources
2. **Access levels:**
   - `view_only: false` - Full access (query and modify)
   - `view_only: true` - Read-only access (query only)
3. **Users access data sources** through their group memberships

## Access Control Logic

### Group-Based Access

Objects (queries, dashboards) are accessible based on group membership:

**Implementation** (from `/home/daytona/workspace/source/redash/permissions.py:34-49`):

```python
def has_access_to_groups(obj, user, need_view_only):
    groups = obj.groups if hasattr(obj, "groups") else obj
    
    if "admin" in user.permissions:
        return True
    
    matching_groups = set(groups.keys()).intersection(user.group_ids)
    
    if not matching_groups:
        return False
    
    required_level = 1 if need_view_only else 2
    
    group_level = 1 if all(flatten([groups[group] for group in matching_groups])) else 2
    
    return required_level <= group_level
```

**Access logic:**
1. Admins have access to everything
2. User must belong to at least one group that has access
3. For modify access, user needs at least one group with non-view-only access
4. For view access, any group access (including view-only) is sufficient

### Combined Access Check

The system checks both API key access and group-based access:

**Implementation** (from `/home/daytona/workspace/source/redash/permissions.py:17-21`):

### API Key Access

API keys can access objects they're associated with:

**Implementation** (from `/home/daytona/workspace/source/redash/permissions.py:24-31`):

```python
def has_access_to_object(obj, api_key, need_view_only):
    if obj.api_key == api_key:
        return need_view_only
    elif hasattr(obj, "dashboard_api_keys"):
        # check if api_key belongs to a dashboard containing this query
        return api_key in obj.dashboard_api_keys and need_view_only
    else:
        return False
```

**Rules:**
- Query API keys can only view (not modify) their associated query
- Dashboard API keys can view queries embedded in the dashboard

```python
@classmethod
def all_queries(cls, group_ids, user_id=None, include_drafts=False, include_archived=False):
    query_ids = (
        db.session.query(distinct(cls.id))
        .join(DataSourceGroup, Query.data_source_id == DataSourceGroup.data_source_id)
        .filter(Query.is_archived.is_(include_archived))
        .filter(DataSourceGroup.group_id.in_(group_ids))
    )
    queries = (
        cls.query.options(
            joinedload(Query.user),
            joinedload(Query.latest_query_data).load_only("runtime", "retrieved_at"),
        )
        .filter(cls.id.in_(query_ids))
    )
    
    if not include_drafts:
        queries = queries.filter(or_(Query.is_draft.is_(False), Query.user_id == user_id))
    return queries
```

**Access rules:**
1. User must have access to the query's data source (via group membership)
2. Published queries are visible to all users with data source access
3. Draft queries are only visible to their creator (unless including drafts)

## Dashboard Access Control

Dashboards have complex access control based on widget data sources:

**Implementation** (from `/home/daytona/workspace/source/redash/models/__init__.py:1145-1161`):

```python
@classmethod
def all(cls, org, group_ids, user_id):
    query = (
        Dashboard.query.options(joinedload(Dashboard.user).load_only("id", "name", "details", "email"))
        .distinct(cls.lowercase_name, Dashboard.created_at, Dashboard.slug)
        .outerjoin(Widget)
        .outerjoin(Visualization)
        .outerjoin(Query)
        .outerjoin(DataSourceGroup, Query.data_source_id == DataSourceGroup.data_source_id)
        .filter(
            Dashboard.is_archived.is_(False),
            (DataSourceGroup.group_id.in_(group_ids) | (Dashboard.user_id == user_id)),
            Dashboard.org == org,
        )
    )
    
    query = query.filter(or_(Dashboard.user_id == user_id, Dashboard.is_draft.is_(False)))
    return query
```

**Access rules:**
1. User must have access to at least one data source used in the dashboard's widgets, OR
2. User is the dashboard owner
3. Draft dashboards are only visible to their creator

```python
def get(self, object_type, object_id):
    model = get_model_from_type(object_type)
```

## Best Practices

1. **Use groups for data source access:** Grant data source access to groups, not individual users
2. **Minimize admin users:** Only grant admin permissions to users who truly need them
3. **Use view-only for read-only access:** Prevent accidental data modifications
4. **Object permissions for collaboration:** Use access permissions to share specific queries/dashboards
5. **Regular access reviews:** Periodically audit group memberships and data source access
6. **Principle of least privilege:** Grant only the minimum permissions needed
7. **Default group security:** Configure your default group with appropriate permissions for new users
8. **Monitor permission changes:** Review permission grant/revoke events in audit logs

## Common Permission Scenarios

### Scenario 1: Read-Only Data Source Access

To give a group read-only access to a data source:

1. Create or identify the group
2. Add data source to group with `view_only: true`
3. Users can query the data source but not modify it

### Scenario 2: Sharing a Query with a Specific User

To share a query with a user who doesn't have data source access:

1. Grant `view` access via object permissions: `POST /api/queries/{id}/acl`
2. User can view and execute the query
3. User cannot modify the query (unless `modify` access is also granted)

### Scenario 3: Department-Based Access

To implement department-based access control:

1. Create groups for each department (Sales, Engineering, etc.)
2. Grant each group access to relevant data sources
3. Assign users to their department groups
4. Users automatically inherit access to their department's data sources

### Scenario 4: External Stakeholder Access

To give external stakeholders limited access:

1. Create a dedicated group with minimal permissions
2. Remove sensitive permissions (e.g., `create_query`, `list_users`)
3. Grant view-only access to specific data sources
4. Use object permissions to share specific dashboards
5. Consider using query/dashboard API keys instead for very limited access
