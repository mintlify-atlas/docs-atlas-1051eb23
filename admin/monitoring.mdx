## Metrics with StatsD

Redash can send metrics to a StatsD server for monitoring and alerting.

### StatsD Configuration

```bash
REDASH_STATSD_HOST=127.0.0.1
REDASH_STATSD_PORT=8125
REDASH_STATSD_PREFIX=redash
REDASH_STATSD_USE_TAGS=false  # InfluxDB/Telegraf style tags
```

### Available Metrics

#### Request Metrics

```
redash.requests.{endpoint}.{method}  # Timing in milliseconds
```

Tracks:
- Request duration
- Endpoint and HTTP method
- Automatically collected for all requests

#### Database Metrics


Tracks:
- Database query duration

Queues:

### Grafana Dashboard

Use Grafana with StatsD to visualize Redash metrics:

1. **Install Graphite or InfluxDB** to collect StatsD metrics
2. **Configure Redash** to send metrics
3. **Create Grafana dashboards** with panels for:
   - Request rate and latency
   - Query execution times
   - Job queue sizes
   - Database performance
   - Error rates

Example queries:

```sql
-- Request rate (InfluxDB)
SELECT mean("value") 
FROM "redash.requests.query.get" 
WHERE time > now() - 1h 
GROUP BY time(1m)

-- Failed jobs (InfluxDB)
SELECT sum("value") 
FROM "redash.rq.jobs.failed.queries" 
WHERE time > now() - 24h 
GROUP BY time(1h)
```

## Logging

### Log Configuration

```bash
REDASH_LOG_LEVEL=INFO              # DEBUG, INFO, WARNING, ERROR, CRITICAL
REDASH_LOG_STDOUT=false            # Log to stdout instead of file
REDASH_LOG_PREFIX=""               # Prefix for log messages
REDASH_LOG_FORMAT="[%(asctime)s][PID:%(process)d][%(levelname)s][%(name)s] %(message)s"
```

### RQ Worker Job Logs

Customize logging for background jobs:

```bash
REDASH_RQ_WORKER_JOB_LOG_FORMAT="[%(asctime)s][PID:%(process)d][%(levelname)s][%(name)s] job.func_name=%(job_func_name)s job.id=%(job_id)s %(message)s"
```

### Log Levels

- **DEBUG** - Detailed diagnostic information
- **INFO** - General informational messages
- **WARNING** - Warning messages for potentially harmful situations
- **ERROR** - Error messages for failures
- **CRITICAL** - Critical failures that may cause shutdown

### Viewing Logs

#### Docker Deployment

```bash
# View server logs
docker logs redash-server

# Follow logs in real-time
docker logs -f redash-server

# View worker logs
docker logs redash-worker

# View scheduler logs
docker logs redash-scheduler

# Filter for errors only
docker logs redash-server 2>&1 | grep ERROR
```

#### Systemd Service

```bash
# View logs
sudo journalctl -u redash-server

# Follow logs
sudo journalctl -u redash-server -f

# View logs from last hour
sudo journalctl -u redash-server --since "1 hour ago"
```

### Structured Logging

Redash uses Python's logging module. Key loggers:

- `redash` - Main application logger
- `metrics` - Metrics logging
- `rq.worker` - RQ worker logs
- `sqlalchemy.engine` - Database queries (when enabled)

### Metrics Logger

Enable detailed metrics logging:

```python
import logging
logging.getLogger('metrics').setLevel(logging.DEBUG)
```

Logs include:
- Request duration
- Database query duration
- Table and action for each query

## Background Job Monitoring

### RQ Dashboard

Redash uses RQ (Redis Queue) for background jobs. Monitor queues:

```python
from redash.monitor import rq_status

status = rq_status()
print(status['queues'])  # Queue sizes and running jobs
print(status['workers']) # Worker information
```

### Queue Status

Check queue sizes:

```python
from redash.monitor import get_queues_status

queues = get_queues_status()
for queue_name, info in queues.items():
    print(f"{queue_name}: {info['size']} jobs")
```

### Worker Status

Monitor RQ workers:

```python
from redash.monitor import rq_workers

for worker in rq_workers():
    print(f"Worker: {worker['name']}")
    print(f"  State: {worker['state']}")
    print(f"  Current Job: {worker['current_job']}")
    print(f"  Successful: {worker['successful_jobs']}")
    print(f"  Failed: {worker['failed_jobs']}")
```

## Alerting

### Failed Query Alerts

Enable email notifications for failed scheduled queries:

```bash
# Via organization settings
PATCH /api/settings/organization
{
  "send_email_on_failed_scheduled_queries": true
```

Configuration:

```bash
REDASH_SEND_FAILURE_EMAIL_INTERVAL=60        # Minutes between emails
REDASH_MAX_FAILURE_REPORTS_PER_QUERY=100     # Max reports per query
```

## Performance Metrics

### Key Metrics to Track

#### Application Health

- Request rate (requests/second)
- Response time (p50, p95, p99)
- Error rate (4xx, 5xx responses)
- Active user sessions

#### Query Execution

- Query execution time
- Query success/failure rate
- Number of queries per data source
- Query result cache hit rate

#### Background Jobs

- Queue depth (jobs waiting)
- Job execution time
- Job failure rate
- Worker utilization

#### Resources

- CPU usage
- Memory usage
- Database connections
- Redis memory usage
- Disk space (query results)

## Troubleshooting

### High Memory Usage

1. Check Redis memory: `redis_used_memory_human`
2. Review query result sizes
3. Enable query results cleanup
4. Reduce query results TTL

### Slow Queries

1. Check database query logs
2. Review data source performance
3. Optimize query SQL
4. Add database indexes

### Queue Backlog

1. Check worker status
2. Increase number of workers
3. Review failed jobs
4. Increase job timeout limits

### High Error Rate

1. Review application logs
2. Check data source connectivity
3. Monitor external dependencies
4. Review recent configuration changes

## Best Practices

### Regular Monitoring

- Set up automated alerts for critical metrics

- Document normal baseline values

### Log Management

- Rotate logs regularly

- Monitor resource trends
- Plan for growth
- Test scaling strategies
- Document resource requirements

### Incident Response

- Define incident severity levels
