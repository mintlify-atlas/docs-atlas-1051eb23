Data sources are the foundation of Redash. This guide explains how to connect to your databases, APIs, and other data sources.

## Adding a Data Source

<Steps>
  <Step title="Navigate to Data Sources">
    Click on **Settings** (gear icon) in the top navigation, then select **Data Sources**.
  </Step>
  
  <Step title="Create New Data Source">
    Click the **New Data Source** button.
  </Step>
  
  <Step title="Select Type">
    Choose your data source type from the list of 50+ supported options.
  </Step>
  
  <Step title="Configure Connection">
    Fill in the required configuration fields:
    - **Name**: A descriptive name for this connection
    - **Connection details**: Host, port, credentials, etc.
    - **Advanced options**: SSL, timeout, and other settings
  </Step>
  
  <Step title="Test Connection">
    Click **Test Connection** to verify your settings work correctly.
  </Step>
  
  <Step title="Save">
    Click **Create** to save the data source.
  </Step>
</Steps>

## Common Configuration Fields

While each data source has specific requirements, these fields are common across many:

### Authentication

- **Username/User** - Database username
- **Password** - Database password (stored securely)
- **API Key/Token** - For API-based sources
- **Service Account** - JSON credentials for cloud services

### Connection Details

- **Host** - Server hostname or IP address
- **Port** - Port number (defaults vary by database)
- **Database/Schema** - Database or schema name
- **Connection String** - Full connection string (some sources)

### Advanced Options

- **SSL/TLS** - Enable encrypted connections
- **SSL Mode** - Certificate verification level
- **Timeout** - Query timeout in seconds
- **Connection Pool** - Maximum concurrent connections

## SSH Tunneling

Redash supports SSH tunneling for secure connections to databases behind firewalls.

### Configuring SSH Tunnel

1. Enable SSH tunnel in data source settings
2. Provide SSH server details:
   - SSH Host
   - SSH Port (default: 22)
   - SSH Username
   - SSH Private Key or Password

3. Configure the actual database connection to use `localhost` as the host

<Note>
SSH tunneling routes your database connection through an SSH server, allowing access to databases not directly exposed to the internet.
</Note>

## Security Best Practices

<AccordionGroup>
  <Accordion title="Use Read-Only Credentials">
    Create database users with read-only permissions for Redash. This prevents accidental data modification through queries.
    
    ```sql
    -- PostgreSQL example
    CREATE USER redash_readonly WITH PASSWORD 'secure_password';
    GRANT CONNECT ON DATABASE mydb TO redash_readonly;
    GRANT USAGE ON SCHEMA public TO redash_readonly;
    GRANT SELECT ON ALL TABLES IN SCHEMA public TO redash_readonly;
    ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT ON TABLES TO redash_readonly;
    ```
  </Accordion>
  
  <Accordion title="Enable SSL/TLS">
    Always use encrypted connections when connecting to databases over the network.
    
    - Upload SSL certificates if required
    - Set SSL mode to `require` or `verify-full`
    - Use cloud provider's SSL certificates for managed databases
  </Accordion>
  
  <Accordion title="Network Security">
    Implement network-level security:
    
    - Whitelist Redash server IP addresses
    - Use VPC peering for cloud deployments
    - Enable SSH tunneling for databases behind firewalls
    - Use VPN connections when appropriate
  </Accordion>
  
  <Accordion title="Credential Management">
    - Store credentials securely in Redash's encrypted storage
    - Rotate passwords regularly
    - Use IAM roles instead of static credentials (AWS)
    - Use service accounts with minimal permissions (GCP)
  </Accordion>
</AccordionGroup>

## Testing Your Connection

After configuring a data source:

1. **Test Connection** - Verifies basic connectivity
2. **Create a test query** - Run a simple query to confirm:
   ```sql
   SELECT 1
   ```
3. **Check schema** - Verify tables appear in the schema browser

## Connection Troubleshooting

<AccordionGroup>
  <Accordion title="Connection Timeout">
    **Symptoms**: "Connection timeout" or "Failed to connect" errors
    
    **Solutions**:
    - Verify host and port are correct
    - Check firewall rules allow connections from Redash
    - Increase timeout setting if database is slow to respond
    - Verify database server is running
  </Accordion>
  
  <Accordion title="Authentication Failed">
    **Symptoms**: "Access denied" or "Invalid credentials" errors
    
    **Solutions**:
    - Double-check username and password
    - Verify user has necessary permissions
    - Check if IP-based authentication is configured correctly
    - Ensure password doesn't contain special characters that need escaping
  </Accordion>
  
  <Accordion title="SSL/TLS Errors">
    **Symptoms**: "SSL connection error" or "Certificate verification failed"
    
    **Solutions**:
    - Try different SSL modes (disable, allow, prefer, require)
    - Upload correct SSL certificate
    - Check certificate hasn't expired
    - Verify hostname matches certificate
  </Accordion>
  
  <Accordion title="Database Not Found">
    **Symptoms**: "Database does not exist" errors
    
    **Solutions**:
    - Verify database name is spelled correctly
    - Check if database requires schema specification
    - Ensure user has access to the specified database
  </Accordion>
</AccordionGroup>

## Managing Data Sources

### Editing a Data Source

1. Go to **Settings** → **Data Sources**
2. Click on the data source name
3. Modify settings as needed
4. Test and save changes

<Warning>
Changing data source settings may affect existing queries. Test thoroughly before saving.
</Warning>

### Deleting a Data Source

1. Navigate to the data source settings
2. Click **Delete** at the bottom
3. Confirm deletion

<Warning>
Deleting a data source will make all queries using it non-functional. This action cannot be undone.
</Warning>

## Data Source Groups

Data sources can be assigned to groups to control access:

1. Edit the data source
2. Under **Groups**, select which groups can access it
3. Save changes

<Note>
Users can only query data sources accessible to their groups.
</Note>

## Environment Variables

Some data sources support configuration via environment variables:

- `REDASH_DATABASE_URL` - Main metadata database
- `REDASH_REDIS_URL` - Redis connection
- Custom environment variables per data source type

Refer to specific data source documentation for environment variable options.
